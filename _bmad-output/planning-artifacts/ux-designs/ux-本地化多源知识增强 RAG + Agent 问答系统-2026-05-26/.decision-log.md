## Decisions

1. Treat Open WebUI as the first-stage interaction shell, not as the authority for enterprise RAG policy.
   - Rationale: PRD and AGENTS.md require `tenant_id`, `user_id`, `acl`, citation, retrieval logs, eval, and audit to be enforced in backend services. Open WebUI is useful as an early chat and model-entry surface, but authority belongs to the project backend.

2. Keep business fine-tuning additive and extension-first.
   - Rationale: MVP should not fork a complex custom frontend before ingestion, retrieval, citation, RBAC, eval, and observability are proven. UI deltas should be expressed as citation rendering, scope signals, job state, diagnostics, and controlled tool events.

3. Make provenance the primary UX object.
   - Rationale: The core user trust moment is not "the answer appeared"; it is "the answer is grounded, authorized, and traceable." Citation, document version, page/chunk, request id, and retrieval stage metadata must be exposed through predictable patterns.

4. Separate employee-facing clarity from admin-facing diagnostics.
   - Rationale: Employee users need answer, source, and current access scope. Engineers and admins need dense/sparse top_k, RRF, rerank score, latency, error code, index status, and eval drift. Mixing these creates cognitive overload and can leak sensitive metadata.

5. Agent UI remains phase-gated behind Tool Registry.
   - Rationale: The project rules prohibit unbounded agent execution. The UX may describe tool-call events, but production UI must only show Agent flows after schema, permission, timeout, rate limit, max_steps, max_tool_calls, and audit log exist.

## Assumptions

- Open WebUI is used as a hosted/local web surface integrated with this backend, rather than vendoring and deeply rewriting Open WebUI source in MVP.
- Custom citation chips may require either an Open WebUI extension point or a small companion management surface. If unavailable, the MVP fallback is structured markdown citations plus a backend source-detail URL.
- Enterprise SSO is deferred; MVP uses development/test simulated AuthContext plus a lightweight JWT adapter, both producing the same backend AuthContext DTO.
- The first design target is responsive desktop web, with mobile usable for read/query/review but not optimized for high-volume administration.
