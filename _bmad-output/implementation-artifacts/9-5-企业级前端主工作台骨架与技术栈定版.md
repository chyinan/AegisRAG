# Story 9.5: 企业级前端主工作台骨架与技术栈定版

Status: review

生成时间：2026-06-10T10:59:23+08:00

## Story

As a 平台负责人,
I want 一个以本项目后端能力为中心的企业级前端主工作台,
so that 用户能在第一屏直接感知 citation、权限、诊断和治理能力，而不是只看到通用聊天壳。

## Acceptance Criteria

1. **产品边界从 Open WebUI 唯一入口切换为自有主界面**
   - Given 9.4 已完成真实 LLM provider 和 `/chat`、`/v1/chat/completions` 端到端闭环
   - When 实现主前端骨架
   - Then README、docs 和 demo 文案必须明确：自有前端工作台是主产品界面，Open WebUI 是兼容入口和演示入口
   - And 不得删除或弱化现有 `/sidecar`、`/governance` 的安全 drilldown 能力；它们在迁移到主工作台前仍是可验证治理入口

2. **前端技术栈定版并可本地运行**
   - Given 当前 `apps/web` 只包含静态 `sidecar` 与 `governance` 页面
   - When 新增主前端应用
   - Then 默认选择 React + Next.js + TypeScript，并在 `apps/web/package.json` 固化前端 scripts
   - And 采用 App Router、TypeScript strict、ESLint/Next lint、Vitest 或等价单测、Playwright 或等价浏览器检查
   - And 数据获取默认使用 TanStack Query，基础无障碍 primitives 可使用 Radix primitives，图标优先使用 `lucide-react`
   - And 不引入小众、弱维护或与 Python/uv monorepo 协作成本高的框架
   - And `apps/web/sidecar`、`apps/web/governance` 路径继续由 FastAPI 静态路由服务，不被 Next.js 构建破坏

3. **首屏是可工作的企业知识工作台，不是营销页或通用 demo**
   - Given 用户打开新主前端
   - When 初始页面完成加载
   - Then 第一屏必须展示企业知识工作台三块核心区域：会话/问题区、证据与 citation 区、诊断/治理上下文区
   - And 桌面默认布局应支持左侧会话/上下文导航、中间 RAG 会话主区、右侧证据/诊断面板
   - And 移动端降级为单列主区 + bottom sheet 或 drawer，不丢失 citation drilldown、request_id copy、权限拒绝和 no-answer 链路
   - And 不得实现营销 hero、通用聊天壳、装饰性渐变页面或只靠大卡片堆叠的布局

4. **后端安全合同由 typed API client 承接**
   - Given 后端已有 `/chat`、`/chat/stream`、`/query`、`/query/stream`、`/sources/resolve`、`/diagnostics/resolve`、`/documents/review`、`/eval/reports`、`/audit/logs`、`/review/items`、`/agent/run`、`/v1/models`
   - When 前端读取或提交业务数据
   - Then 必须通过集中 typed API client 或 server-side proxy/rewrites 调用后端，统一处理 `{request_id,data,error,metadata}` envelope、SSE events 和 structured errors
   - And 前端不得判断权限、补造 citation、推断未授权资源存在性、拼接 prompt、读取任意本地文件或直接调用模型 provider
   - And 浏览器可见配置只能包含 API base URL 等非密钥信息；不得将 bearer token、service token、API key、provider key 写入 `NEXT_PUBLIC_*`

5. **会话与 SSE 流式体验保持 RAG 安全语义**
   - Given 用户在主工作台提交问题
   - When `/chat/stream` 或 `/query/stream` 返回 `token`、`citation`、`tool_call`、`tool_result`、`error`、`final`
   - Then UI 必须逐步渲染 token，并在 `citation` 或 `final` 后锁定 citation 列表
   - And `Copy answer with citations` 或等价复制动作只能在 terminal `final` 后启用
   - And no-answer 是正式成功状态，文案遵循“无法从当前授权资料确认”，并提供 request_id、查看检索范围、上传或联系管理员的安全下一步
   - And provider 输出中的来源文本不被信任为 citation；citation 只能来自后端事件或 response data

6. **Source Evidence 与 Diagnostics 嵌入主工作台**
   - Given 回答包含 citation 或 evidence link
   - When 用户打开 evidence 面板
   - Then 前端必须重新调用 `/sources/resolve` 做二次授权，只展示后端返回的授权片段和安全 source metadata
   - And denied、not found、soft deleted、inactive version、ACL mismatch 使用同一安全失败形态，不暴露目标资源是否存在，也不保留上一次授权 excerpt
   - Given 用户复制或输入 request_id/trace_id
   - When 打开 Diagnostics 面板
   - Then UI 调用 `/diagnostics/resolve` 展示 safe timeline：top_k、result_count、highest rerank score、citation_count、latency、failure_stage、error_code 和 next_steps
   - And 不显示 raw query、chunk content、prompt、SQL、vectors、embeddings、provider payload 或 secrets

7. **治理入口作为主工作台稳定导航，而不是孤立静态页**
   - Given Epic 8 已实现 Document Review、Source Evidence、Retrieval Diagnostics、Eval Evidence、Audit Explorer、Review Queue
   - When 主工作台定义信息架构
   - Then 必须提供这些治理入口的稳定导航或面板占位，并明确哪些在本 story 直接接入、哪些保留跳转到现有 `/governance`
   - And 任何未接入入口必须显示后端能力边界和安全空状态，不得用假数据伪装真实治理结果
   - And tool events 必须以受控事件行展示，关联 `agent_run_id`、`tool_name`、status、latency、error_code、request_id、trace_id，不展示 raw arguments/output

8. **视觉与交互符合企业级工作台气质**
   - Given 主前端需要 UI 规范
   - When 实现 design tokens 和基础组件
   - Then 使用克制、高密度、可扫描的企业工作台风格：清晰表格/列表、分栏布局、稳定状态层级、8px 或更小圆角、少量语义色
   - And citation、权限拒绝、tool event、diagnostics、review item 必须有一致视觉层级和非纯颜色状态表达
   - And 不使用营销式 hero、花哨渐变、装饰 orb、过大字号、通用 AI 聊天壳或整页浮动卡片
   - And 长 document_id、version_id、chunk_id、request_id、trace_id、文件名和 URL 必须换行或截断，并提供完整值复制方式

9. **认证上下文与权限失败只由后端事实驱动**
   - Given 本地开发仍可能使用 dev/test auth headers 或 JWT
   - When 前端发送请求
   - Then 请求必须透传用户输入的 Authorization/dev headers 或通过同源 proxy 转发，不在前端扩大 tenant、roles、permissions
   - And 缺少 AuthContext 时 UI 显示结构化 `AUTH_CONTEXT_REQUIRED` 或等价错误，不展示任何知识库、文档名或历史片段
   - And 前端 scope 选择只能收窄后端返回的授权范围，不能扩大权限或构造 tenant_id

10. **测试、文档和 Docker/本地启动闭环**
    - Given story 完成
    - When 运行验证
    - Then 必须提供前端 unit/component tests、关键浏览器检查、类型检查、lint，以及不真实调用外部 LLM 的 API mock/MSW 或等价测试边界
    - And README、`docs/operations/local-development.md` 和相关 demo docs 必须更新主前端启动方式、环境变量、Open WebUI 兼容边界、当前能力和限制
    - And 如果新增 Docker web service 或 compose profile，应有 healthcheck 或明确本地开发命令；如果暂不纳入 compose，README 必须说明原因和后续故事边界

### Backend API Contracts Relevant To Main Workbench

- Chat and query:
  - `POST /chat`, `POST /chat/stream`
  - `POST /query`, `POST /query/stream`
  - Stream formatter emits project SSE events from `packages/rag/streaming.py`.
  - `/chat/stream` is POST SSE; use fetch streaming parser, not native EventSource.

- Evidence and diagnostics:
  - `POST /sources/resolve`
  - `POST /diagnostics/resolve`
  - Evidence excerpts require backend re-authorization. Frontend must not keep stale excerpts after denial or failed lookup.

- Governance:
  - `GET /documents/review`
  - `GET /documents/{document_id}/review`
  - `GET /documents/{document_id}/versions/{version_id}/review`
  - `GET /documents/{document_id}/versions/{version_id}/status`
  - `GET /eval/reports`
  - `GET /eval/reports/{report_filename}`
  - `GET /audit/logs`
  - `POST /audit/export`
  - `GET/POST /review/items` and review status/eval-candidate routes
  - `POST /agent/run`
  - These should be wrapped through typed clients and safe allowlists before any UI rendering.

### Previous Story Intelligence

- Story 8.1-8.7 established governance workbench safety: frontend displays backend-confirmed facts only and must not decide authorization, infer citations, or expose raw prompts/chunks/provider payloads.
- Story 9.1 established Open WebUI evidence links. Main workbench should accept the same safe identifiers and evidence URLs.
- Story 9.2 established safe tool event summaries. Main workbench tool rows must show safe summaries, not raw tool input/output.
- Story 9.3 established Open WebUI function/tool bridge. Main frontend must not call arbitrary tools or create a second tool execution path.
- Story 9.4 established real LLM provider behind `LLMProvider` and `/chat`. Main frontend uses backend chat endpoints; it never calls provider APIs or uses provider keys.

### Architecture Compliance Requirements

- `apps/web` may contain the frontend app, but Python backend route files stay under `apps/api`.
- Frontend business calls go through backend endpoints. No frontend provider SDKs, no direct OpenAI/Qwen/DeepSeek calls, no prompt construction.
- AuthContext truth remains in backend. Frontend can display scope returned by backend but cannot create tenant/role/permission authority.
- Existing Python tests and static sidecar tests remain part of the verification set.
- README synchronization is required because this story changes project capability and product positioning.

### Frontend Design Requirements

- Purpose: enterprise RAG command workbench for platform负责人、企业员工、知识库管理员、交付顾问、审计/工程人员.
- Tone: restrained, dense, audit-focused, production operations quality. Think “knowledge operations console”, not “AI chatbot demo”.
- First viewport signal: working knowledge workbench with active query/composer, current scope, citation/evidence status, request_id/trace_id, diagnostics entry.
- Layout:
  - Desktop `>=1200px`: left navigation/history/context, center chat/work item, right evidence/diagnostics.
  - Tablet `768-1199px`: collapsible nav; evidence drawer overlays but does not block composer.
  - Mobile `<768px`: single-column chat; evidence/diagnostics bottom sheet; governance lists become summary + detail.
- Visual constraints:
  - Cards only for repeated items, modals, tool rows and framed tools; no cards inside cards.
  - Radius max 8px unless existing component requires less.
  - No hero, no marketing split layout, no decorative gradients/orbs, no stock-like empty state art.
  - Text must not overflow buttons/cards/panels. Use `overflow-wrap:anywhere`, copy affordances and tooltips for long IDs.
  - Use icons for copy/open/close/filter/search/diagnostics/tool event actions.
