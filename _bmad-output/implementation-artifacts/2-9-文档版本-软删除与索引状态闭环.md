# Story 2.9: 文档版本、软删除与索引状态闭环

Status: done

生成时间：2026-06-06T18:59:21+08:00

## Story

As a 知识库管理员,
I want 看到文档版本、索引状态和删除状态,
so that 知识库不会出现不可追踪的覆盖、误删或半索引状态。

## Acceptance Criteria

1. **同一文档再次上传会创建新版本，不覆盖旧版本**
   - Given 管理员上传文件时提供已有 `document_id`
   - When 上传内容 checksum、`source_uri` 或 `metadata.source_version` 与现有版本不同
   - Then 系统在同一 tenant/document 下创建新的 `version_id`
   - And 旧 `document_versions`、`chunks`、`embedding_jobs`、`vector_records` 不被静默覆盖或硬删除
   - And `/upload` 仍立即返回 `document_id`、新 `version_id`、`job_id`、`status=uploaded`，不等待 parser/chunker/embedding/vector indexing

2. **索引成功后版本进入 `retrieval_ready`**
   - Given ingestion、chunk persistence、embedding 和 vector indexing 全部成功
   - When embedding worker 完成当前 version 的安全索引摘要校验
   - Then `document_versions.status` 变为 `retrieval_ready`
   - And 对应 `documents.status` 反映最新非删除版本的可检索状态
   - And 可查询 chunk 数量、embedding provider/model/version/dim、vector count、索引状态、错误码、request_id/trace_id 摘要

3. **软删除默认使文档或版本不可检索**
   - Given 管理员删除整个 document 或指定 version
   - When 调用删除 API/application service
   - Then 默认软删除，设置 `status=deleted` 和 `deleted_at`
   - And 关联 active chunks 被软删除
   - And 调用 `VectorStore.delete_by_document(document_id, version_id, tenant_id=...)` 标记向量记录删除，使后续 search 默认不返回
   - And 删除操作记录审计日志，包含 request_id、trace_id、tenant_id、user_id、document_id、version_id、deleted_count、status、error_code，不记录文档正文、向量、secret 或本机绝对路径

4. **权限、tenant 和状态边界不可绕过**
   - Given 请求缺少 AuthContext、tenant 或 `document:manage`
   - When 上传新版本、查询版本状态或删除文档/版本
   - Then service 拒绝请求并返回稳定错误码
   - And route 层只做 schema/AuthContext/service 调用/响应封装
   - And 任何跨租户 document/version/chunk/vector 操作都返回拒绝或 not found，不泄露资源存在性

### Architecture Requirements

- 本 story 横跨 API Layer、Application Service Layer、Domain/DTO、Storage Layer 和 VectorStore port 调用；不要把状态机写进 FastAPI route。
- `packages/data` 负责文档、版本、chunk、job 状态治理；`packages/vectorstores` 只负责 vector records 的统一写入/查询/删除契约。
- `tenant_id`、`user_id`、`document_id`、`version_id`、`chunk_id`、`acl` 必须从 DTO、repository、service、audit/log 贯穿。
- 删除和状态查询必须在后端执行权限校验；不要让 LLM、prompt 或前端判断权限。
- SQLAlchemy model 不能穿透到 domain/application 返回值；API 返回 DTO/Pydantic schema。

### Implementation Boundaries

- 不要实现 BM25/PostgreSQL full text sparse retrieval、RRF、rerank、retrieve API、context packing、RAG answer 或 citation extraction；这些属于 Epic 3/4。
- 不要 hard delete documents、versions、chunks、vector records 或 object storage raw files。
- 不要新增 Milvus/FAISS production adapter；继续使用现有 fake/pgvector 边界。
- 不要把 `retrieval_ready` 写进 prompt 或前端推断；后端状态机和 repository 是真相来源。
- 不要把完整向量、chunk 正文、provider raw response、API key、access token、本机绝对路径写入 logs/audit/metadata/API response。

### Latest Technical Information

- SQLAlchemy 2.0 官方 session 文档说明 flush 失败后同一 Session 需要显式 rollback，且 AsyncSession 并发模型是每个 task 使用独立 session；本 story 的 repository/service 失败路径必须继续 rollback，避免半状态事务污染。Source: https://docs.sqlalchemy.org/en/20/orm/session_basics.html
- SQLAlchemy asyncio 文档也强调 flush failure 后需要 `Session.rollback()` 才能脱离 partial rollback 状态；软删除和 retrieval_ready 更新应沿用现有 `StorageError -> rollback` 模式。Source: https://docs.sqlalchemy.org/20/orm/extensions/asyncio.html
- pgvector README 展示可用 HNSW index、partial index 和 SQL filter；2.9 删除后必须依赖 `status/deleted_at` 查询过滤和 `VectorStore.delete_by_document`，不要指望调用方在检索后过滤 deleted 结果。Source: https://github.com/pgvector/pgvector
