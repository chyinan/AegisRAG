# Story 1.5: 最小数据库迁移与基础治理表

Status: done

生成时间：2026-05-27T11:07:09+08:00

## Story

As a 平台工程师,
I want Alembic 管理基础身份、租户和审计表,
so that 后续业务数据能够从第一天开始携带治理字段。

## Acceptance Criteria

1. **空数据库可通过 Alembic 创建基础治理表**
   - Given 数据库为空
   - When 运行 Alembic migration
   - Then 创建 `tenants`、`users`、`roles`、基础用户角色关系表和 `audit_logs`
   - And 所有表包含 `id`、`created_at`、`updated_at`

2. **关键业务表包含治理字段且不依赖自动建表**
   - Given 关键业务表被创建
   - When 检查 schema
   - Then 表必须包含适用的 `tenant_id`、`created_by`、`status`
   - And migration 不依赖应用启动时自动建表

3. **Storage smoke test 返回类型化 DTO**
   - Given 测试环境运行 storage smoke test
   - When 插入一条租户、用户和审计事件
   - Then repository 返回类型化 DTO
   - And domain 层不接收 SQLAlchemy model

### Architecture Requirements

- 本 Story 属于 Storage Layer + Infrastructure Adapter 基础，不属于 API route 或 domain 逻辑。
- API route 仍保持薄层；不要为了验证 DB 在 route 中写 repository 查询或迁移逻辑。
- `packages/*/domain` 不得导入 SQLAlchemy、FastAPI、Redis、MinIO、外部 SDK。
- `packages/common` 保持 framework/infrastructure-free；不要把 SQLAlchemy Declarative Base、engine、session factory 放入 common。
- Alembic migration 是 schema 真相；应用启动不得自动建表。
- Storage model 与 DTO 必须分离。Repository 是 SQLAlchemy model 到 DTO 的边界。
- 审计日志是安全元数据，不是业务全文存储；不要记录 prompt、document chunk、tool raw output、request/response body、token 或 API key。
- 不要提前创建 `documents`、`document_versions`、`chunks`、`embedding_jobs`、`retrieval_logs`、`chat_sessions`、`chat_messages`、`agent_runs`、`tool_calls` 表；这些由后续 story 首次需要时创建。

- Alembic 官方文档支持通过 migration environment 的 `target_metadata` 进行 autogenerate；即使使用 autogenerate，也必须人工复核生成的 constraints、indexes、nullable 和 downgrade。[Source: https://alembic.sqlalchemy.org/en/latest/autogenerate.html]
- Alembic cookbook 提供 asyncio 迁移环境模式；如果采用 async DB URL，`env.py` 应显式处理 async engine/run migrations，而不是在应用运行时自动建表。[Source: https://alembic.sqlalchemy.org/en/latest/cookbook.html]
