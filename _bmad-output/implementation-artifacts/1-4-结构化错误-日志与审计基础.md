# Story 1.4: 结构化错误、日志与审计基础

Status: done

生成时间：2026-05-27T10:12:47+08:00

## Story

As a 平台运维人员,
I want 预期错误、请求日志和审计事件结构化记录,
so that 后续上传、检索、问答和 Agent 行为可以追踪与排障。

## Acceptance Criteria

1. **预期领域错误返回稳定结构化 error**
   - Given application service 抛出预期领域错误
   - When API 捕获错误
   - Then 返回稳定 `error.code`、`error.message`、`error.details`
   - And 不暴露原始异常类名、堆栈或敏感字段

2. **每个 API 请求写出结构化请求日志**
   - Given 任意 API 请求完成
   - When 日志被写出
   - Then 日志包含 `request_id`、`trace_id`、`tenant_id`、`user_id`、`latency`、`error_code`
   - And 不记录 API key、access token、企业机密全文或用户敏感原文

3. **业务服务可通过 audit port 记录审计事件**
   - Given 业务服务记录审计事件
   - When 调用 audit port
   - Then 审计事件包含 action、resource、tenant_id、user_id、status、latency、error_code
   - And 先提供 fake/in-memory audit adapter 供测试使用

### Architecture Requirements

- API route 仍保持薄层：schema、context dependency、service 调用、response envelope；不要在 route 内拼日志字段、构造审计细节或捕获所有异常。
- `packages/common/*` 可以放跨模块 DTO、Protocol、error、logging 和 audit contract，但不得导入 FastAPI、SQLAlchemy、Redis、MinIO、OpenAI/Qwen/DeepSeek SDK 或 vector store SDK。
- API support 文件可以导入 FastAPI/Starlette，但不能声明业务 route；若新增 `apps/api/middleware.py`，同步维护架构边界测试。
- `RequestContext` 是日志、错误和审计的关联主键来源；不要在 middleware、dependency、error handler 各自生成不同 request_id。
- `AuthContext` 是认证事实，不是日志原文。日志和审计只记录 `tenant_id`、`user_id` 以及必要的 roles/permissions 摘要，不记录 bearer token、完整 header 或认证失败 token 内容。
- Audit port 是 application service 可调用的边界；本 Story 只做接口和 fake adapter，不做数据库持久化，避免抢 Story 1.5 的 Alembic/基础治理表工作。
- 结构化日志是生产可观测基础，但 OpenTelemetry span、Prometheus 指标、Grafana dashboard、日志保留清理策略不在本 Story 范围内。
