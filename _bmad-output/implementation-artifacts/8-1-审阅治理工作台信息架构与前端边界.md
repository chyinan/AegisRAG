# Story 8.1: 审阅治理工作台信息架构与前端边界

Status: done

生成时间：2026-06-09T21:30:00+08:00

## Story

As a 平台负责人,
I want 一个能解释企业 RAG 安全能力的审阅治理工作台骨架,
so that 非技术观众也能理解 tenant/RBAC/ACL、citation、diagnostics、eval 和 audit 的价值。

## Acceptance Criteria

1. **工作台有稳定信息架构但不冒充完整管理后台**
   - Given 用户打开审阅治理工作台
   - When 前端加载初始 shell
   - Then 页面必须提供 Document Review、Source Evidence、Retrieval Diagnostics、Eval Evidence、Audit Explorer、Review Queue 六个稳定导航入口
   - And 初始页面清楚展示当前 tenant/user/request scope 的安全占位或后端确认值，但不展示原始 token、完整 query、prompt、chunk 全文或 provider payload
   - And 明确当前 Story 只建设工作台壳、导航、边界和可测试前端契约，不实现完整 CRUD 后台、RBAC 管理器、Grafana 替代品或 Open WebUI fork

2. **工作台只展示后端确认的安全字段**
   - Given 工作台需要展示 source、status、diagnostics、eval、audit 或 review 信息
   - When 前端请求数据
   - Then 所有业务数据必须来自已存在或新增的后端 API/application service
   - And 前端不得判断权限、补造 citation、推断 retrieval result、拼接 prompt、读取本地文件、解析日志文件或直接读取静态 eval report
   - And 每个视图必须定义 allowlist 字段，默认拒绝 `source_uri`、object key、本机路径、prompt、full query、answer text、chunk content、SQL、vectors、embeddings、provider raw response、token、secret、raw exception

3. **复用现有 sidecar/diagnostics 能力，避免重建轮子**
   - Given 当前仓库已有 `/sidecar`、`/sources/resolve`、document status 和 `/diagnostics/resolve`
   - When 实现工作台骨架
   - Then 必须复用或演进 `apps/web/sidecar` 的同源静态资产、safe field allowlists、local/test auth helper、copy/download 安全模式和 JS 行为测试
   - And 不新增 React、Next.js、Vite、Node build pipeline、状态管理库或设计系统，除非实现说明证明现有静态结构无法满足 AC，并同步补测试、docs、Docker/CI 说明

4. **访问、安全失败和数据缺失状态可解释**
   - Given 用户没有某个视图所需权限、目标资源跨 tenant、记录不存在或后端返回安全拒绝
   - When UI 渲染结果
   - Then 使用统一安全失败状态，不暴露目标资源是否存在
   - And 必须清理上一次授权或成功结果，避免 stale data 残留
   - And 页面提供 request_id/trace_id 复制入口和下一步验证命令或 docs 链接，但不泄露内部异常文本

5. **可访问性和响应式行为不退化**
   - Given 工作台在桌面和移动尺寸使用
   - When 用户通过键盘、屏幕阅读器或触控操作
   - Then 满足 WCAG 2.2 AA 基础、焦点恢复、`aria-live`、alert region、非纯颜色状态表达、非 hover-only 操作
   - And 长 document_id、version_id、chunk_id、request_id、trace_id、agent_run_id、tool_call_id 必须安全换行或截断，并提供完整值复制方式

6. **文档、README 和验证闭环同步**
   - Given Story 8.1 完成
   - When 运行 focused checks
   - Then README Build Status、Current Limits、sidecar/governance docs 或 local-development 文档必须说明 Epic 8.1 的入口、能力、限制、安全边界和验证命令
   - And 新增/更新测试覆盖 route、static asset、safe field allowlist、tab/navigation、failure clearing、copy behavior 和 responsive/accessibility contract

### What Must Be Preserved

- Backend `AuthContext`, RBAC, ACL filters, source resolve authorization, diagnostics authorization and audit remain authoritative.
- Open WebUI, sidecar and governance workbench are presentation surfaces only.
- Existing `/sidecar`, `/sources/resolve`, `/diagnostics/resolve`, document status route and OpenAI-compatible routes must not regress.
- Default tests must not require real Open WebUI, browser automation, PostgreSQL, Redis, MinIO, Docker, external network or real LLM/embedding providers.
- Reports, docs and UI output must not include real secrets, tokens, prompts, full query, answer text, full chunks, raw source URI, object keys, local paths, SQL, vectors, embeddings, provider payloads or raw exceptions.
