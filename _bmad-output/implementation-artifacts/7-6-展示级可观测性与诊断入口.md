# Story 7.6: 展示级可观测性与诊断入口

Status: done

生成时间：2026-06-09T00:46:53+08:00

## Story

As a 平台工程师,
I want 演示时能解释一次回答的检索、重排、上下文、生成、citation 和审计状态,
so that 可信 RAG 不只是能回答，还能被复盘。

## Acceptance Criteria

1. **诊断查询受后端权限和租户边界控制**
   - Given 用户完成一次 Open WebUI、`/chat`、`/chat/stream`、`/query`、`/query/stream`、`/retrieve` 或 `/sources/resolve` 请求
   - When 平台工程师按 `request_id` 或 `trace_id` 查询诊断
   - Then 后端必须通过 `AuthenticatedRequestContext` 校验 `audit:read` 或新增的等价诊断权限，并按 `tenant_id` 查询，禁止跨租户诊断
   - And route 只委托应用服务，不能直接查询数据库、拼装业务规则或读取日志文件

2. **诊断摘要只暴露安全聚合字段**
   - Given 诊断命中 retrieval log、RAG/chat audit、source resolve audit 或 request log 摘要
   - When 返回诊断响应
   - Then 能看到安全摘要：`tenant_id`、`user_id`、`request_id`、`trace_id`、action/status、top_k、result_count、highest rerank score、citation count、context item/source counts、generation provider/model/version、token usage counts、event counts、latency、status、error_code
   - And 不包含 full query、query text、answer text、chunk content、prompt、provider raw response、SQL、tsquery/tsvector、vectors、embeddings、tokens/secrets、raw `source_uri`、object key、本机绝对路径或完整异常

3. **失败阶段可复盘且不泄露内部细节**
   - Given retrieval、rerank、context packing、generation、citation、permission、source resolution、audit storage 或 infrastructure 失败
   - When 打开诊断入口
   - Then 响应必须给出稳定 `failure_stage`，枚举值限制为 `retrieval`、`rerank`、`context_packing`、`generation`、`citation`、`permission`、`source_resolution`、`audit`、`infrastructure`、`unknown`
   - And 响应包含可执行下一步验证命令或安全报告位置，例如 focused pytest、eval smoke、walkthrough report 路径
   - And 失败详情只使用稳定 error code、safe counts、request/trace IDs 和阶段名，不暴露资源存在性或内部异常文本

4. **Sidecar Diagnostics 从占位入口升级为安全摘要视图**
   - Given 用户在 sidecar Source Inspector、Job Status 或 Open WebUI/walkthrough 报告中得到 request/trace IDs
   - When 打开 Diagnostics tab
   - Then UI 可输入或自动带入 `request_id`/`trace_id`，调用后端诊断 API，渲染安全摘要、阶段状态、next-step commands 和报告导出按钮
   - And UI 继续不保存 bearer token、dev headers、authorized excerpt、诊断响应或导出报告到 localStorage/sessionStorage/URL
   - And UI 不渲染 full retrieval trace、Grafana 替代品、OpenTelemetry viewer、prompt viewer、chunk viewer 或 provider payload viewer

5. **导出报告 synthetic-safe 且可演示**
   - Given 诊断数据用于演示或调试
   - When 用户复制或下载诊断报告
   - Then 报告只包含 synthetic-safe IDs、计数、latency、status、failure_stage、error_code、next-step commands 和相关 safe report 文件名
   - And 报告文件名或浏览器下载名必须以 `request_id`/`trace_id` 和时间戳构成，不包含 query、answer、document title、source URI、tenant secret 或本机路径
   - And 后端和前端测试覆盖 forbidden fragment 泄露回归

6. **文档、README 和验证闭环同步**
   - Given Story 7.6 实现完成
   - When 运行 focused checks
   - Then 新增或更新 README、`docs/operations/local-development.md`、`docs/demo/source-inspector-sidecar.md` 或专门 diagnostics 文档，说明入口、权限、安全字段、限制和验证命令
   - And README Build Status 从 Story 7.5 更新到 Story 7.6，并把 showcase-grade diagnostics 从 Current Limits 中移除或改为已实现说明
   - And focused tests、`ruff`、`mypy` 和 README expectation tests 通过

### What Must Be Preserved

- Backend AuthContext, RBAC, tenant filtering, ACL, source resolve authorization and audit remain authoritative.
- Diagnostics must never become a privilege escalation path to see another tenant's request, source, query or answer.
- Sidecar is a companion page only. It must not decide tenant, user, role, permission, source visibility, citation validity or report correctness.
- Existing retrieval/RAG/source/eval reports already implement safe summaries. Reuse their fields and conventions; do not duplicate eval runners or demo corpus.
- The implementation must keep deterministic local tests with fake providers. No real LLM, embedding, reranker, Open WebUI container, PostgreSQL, Redis, MinIO, Docker or external network is required for default tests.
- API routes must return shared envelopes and structured errors.
- Logs, audit, diagnostics and exported reports must exclude secrets and protected content by allowlist, not by best-effort string cleanup alone.
