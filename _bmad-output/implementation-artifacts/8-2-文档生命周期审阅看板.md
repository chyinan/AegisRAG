# Story 8.2: 文档生命周期审阅看板

Status: done

生成时间：2026-06-09T23:05:00+08:00

## Story

As a 知识库管理员,
I want 在前端审阅文档、版本、ingestion job 和索引状态,
so that 我可以解释一份文档从上传到 retrieval_ready 的可信处理过程。

## Acceptance Criteria

1. **Document Review 展示后端确认的 tenant-scoped 文档与版本摘要**
   - Given 管理员打开 `/governance` 的 Document Review
   - When 查询文档列表或输入单个 document/version
   - Then UI 展示后端返回的 `document_id`、`version_id`、`source_display_name`、`source_type`、`status`、`created_by`、`created_at`、`updated_at`、`chunk_count`、embedding/indexing 摘要、`request_id`、`trace_id` 和安全错误摘要
   - And 不展示原始 `source_uri`、本机绝对路径、MinIO object key、access token、完整文档正文、chunk content、prompt、SQL、vectors、embeddings、provider raw response 或未授权版本

2. **生命周期时间线使用后端状态契约**
   - Given 文档处于 `uploaded`、`parsing`、`parsed`、`chunking`、`chunked`、`embedding`、`indexing`、`retrieval_ready`、`failed_retryable`、`failed_terminal`、`deleted` 任一状态
   - When 管理员查看生命周期时间线
   - Then UI 展示状态顺序、当前阶段、失败阶段、`attempt_count`、`next_retry_at`、`request_id` 和 `trace_id`
   - And 状态含义来自后端响应、共享常量或被测试锁定的状态映射，前端不得自由扩展或把未知状态当作 working

3. **权限、跨 tenant 和缺失资源统一安全失败**
   - Given 管理员没有权限、目标跨 tenant、文档不存在、版本不存在或后端拒绝
   - When 调用文档审阅 API
   - Then 后端返回统一 structured error
   - And UI 清理旧列表、旧详情和旧时间线，只展示安全失败状态，不暴露目标资源是否存在，并提供 request/trace 复制入口和下一步验证说明

4. **后端 API 保持分层与审计**
   - Given 需要新增文档列表或版本详情能力
   - When 实现 API
   - Then FastAPI route 只负责 schema、AuthContext dependency、service 调用和 envelope
   - And Application Service 执行 `document:manage` 权限、tenant 过滤、repository 调用、安全 DTO 映射、structured audit log 和 error translation
   - And route 不直接访问 SQLAlchemy session、repository、MinIO、vector store、日志文件或前端静态报告

5. **复用 8.1 governance shell，不重建前端栈**
   - Given 当前已有 `/governance`、`apps/web/governance/index.html` 和复用的 `/sidecar/assets/*`
   - When 实现 Document Review 看板
   - Then 默认扩展现有 no-build static HTML/CSS/JS 和 executable JS test runner
   - And 不新增 React、Next.js、Vite、Node build pipeline、状态管理库、Open WebUI fork、Grafana 替代页面或完整 CRUD 后台
   - And `/sidecar` 仍保持 Source Inspector-first，`/governance` 保持 governance-first，现有 Source Evidence、Job Status、Diagnostics 入口不退化

6. **可访问性、响应式和文档验证闭环**
   - Given 看板在桌面、平板和移动尺寸使用
   - When 用户通过键盘、屏幕阅读器或触控操作
   - Then Document Review 支持 tab/焦点顺序、`aria-live`、alert region、非纯颜色状态表达、长 ID 安全换行/截断和完整值复制
   - And README 与 `docs/demo/governance-workbench.md` 同步说明 8.2 的入口、能力、限制、安全边界和验证命令
   - And 新增/更新测试覆盖 backend API/service/repository、safe field allowlist、生命周期状态映射、failure clearing、responsive/accessibility contract 和 README 期望

### Previous Story Intelligence

- Recent review fixes repeatedly caught stale authorized data after failures. Treat stale clearing for document list/detail/timeline as a first-class requirement, not a polish item.

### Implementation Guardrails

- Do not expose `DocumentRecord` or `DocumentVersionRecord` directly to API/UI because they include `source_uri`, `object_key`, `acl` and storage metadata.
- Do not implement unrestricted document enumeration. The list is tenant-scoped, permission-gated and bounded by `limit`; future search/filter expansion must still be backend-authorized.
- Do not let frontend derive `source_display_name` from raw paths or object keys. Use a backend-safe display name.
- Do not use prompt text, eval reports, local log files or static seed artifacts as the source of lifecycle truth.
- Do not add new database tables for review queue, audit explorer or eval evidence in this story. Those belong to Stories 8.5-8.7.
